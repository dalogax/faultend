Using Faultend

Getting started

Faultend sits between your app and your real backend. You point your app at Faultend, create rules that describe what to do with incoming traffic, and watch everything stream through a live traffic viewer. No code changes needed — just swap the API base URL.

cloud

The quickest way to start is app.faultend.com — sign in with Google or GitHub and have a proxy URL ready in under a minute. If you want to run your own instance, jump to the Self-hosting section.

01

Sign in

Go to app.faultend.com and sign in with Google or GitHub. Your account is created automatically on first login.

02

Create a fault server

Click New server, enter a short ID (e.g. my-api), and confirm. Your proxy URL is instantly available at https://my-api.faultend.com.

03

Add a proxy rule

Create a rule that matches all paths (.*) and proxies to your real API. This makes Faultend a transparent pass-through by default.

04

Point your app at Faultend

Change your app's API base URL from https://api.yourapp.com to https://my-api.faultend.com. Traffic flows through Faultend — your app doesn't need to know.

05

Inject failures and observe

Add mock rules to simulate 503s, timeouts, or corrupted responses. Watch your app's behaviour change in real time as requests come through the traffic viewer.

Create a fault server

A fault server is an isolated proxy instance. Each one gets a unique subdomain and its own rules and traffic log. You can have multiple servers — one per project, environment, or test scenario.

Click New server in the top-right of the dashboard. The only required field is the server ID — this becomes part of your proxy URL. IDs must be lowercase alphanumeric with hyphens allowed (e.g. mobile-api, checkout-service).

tip

Use IDs that match your service names — payments-api, auth-service. It makes URLs easy to remember and share with teammates.

Set up routing with proxy rules

Faultend doesn't have a global backend URL. Every request is matched against your rules in priority order. This means you can route different paths to different services, or mock just a subset of endpoints.

A proxy rule needs three things: a path pattern (regex), a priority, and a target URL. Start by creating a catch-all rule that forwards everything to your real API:

proxy pattern.* priority100 targethttps://api.yourapp.com

How priority works

Rules are evaluated from highest priority to lowest. The first rule that matches the incoming request wins. This means specific rules should have higher priority than catch-all rules.

# Example rule stack — evaluated top to bottom Priority 200 — MOCK /api/users/.* → 503 Service Unavailable # specific failure Priority 150 — PROXY /api/payments → https://payments.yourapp.com # payment service Priority 100 — PROXY .* → https://api.yourapp.com # everything else
note

Unmatched requests return 502. If no rule matches, Faultend responds with 502 Bad Gateway. You always need at least one catch-all proxy rule to pass traffic through normally.

Inspect live traffic

Every request that flows through a fault server is logged and shown in the traffic viewer. You can see method, path, status code, latency, and the full request and response headers and body.

Click any row to open the detail panel. It shows the full URL, all request and response headers, the request body, and the response body. Use Copy as cURL to replay the exact request from your terminal.

tip

Traffic is retained for 7 days and capped at 1,000 entries per server. Use the filter bar to search by method, path, or status code.

Mock failures

A mock rule intercepts matching requests and returns a response you define — without touching your real backend. This is how you simulate downtime, broken APIs, or edge-case error responses.

Common scenarios you can test:

503 Service Unavailable — backend is down
Pattern: .* Priority: 200 Action: Mock Status: 503 Body: {"error": "service_unavailable"}
401 Unauthorized — expired token
Pattern: /api/.* Priority: 200 Action: Mock Status: 401 Body: {"error": "token_expired", "message": "Please re-authenticate"}
500 Internal Server Error — specific endpoint
Pattern: /api/checkout Priority: 200 Action: Mock Status: 500 Body: {"error": "internal_error"}
tip

Enable and disable rules with the toggle switch in the rules list — no need to delete and recreate them between test runs.

Inject latency

Any rule — proxy or mock — can add artificial delay before the response is sent. Use this to simulate slow networks, overloaded backends, or timeout scenarios.

Fixed delay
Always adds exactly N milliseconds. Good for simulating a consistently slow API.
Random range
Adds a random delay between a min and max. Good for simulating jittery, unpredictable network conditions.
# Simulate a slow upstream — 800ms to 2000ms random delay Pattern: .* Action: Proxy → https://api.yourapp.com Latency: random 800ms – 2000ms
tip

Combine latency with a mock 200 response to simulate a very slow but technically successful API. This tests whether your app handles slow responses gracefully (spinners, timeouts, cancellation).

One-click mocking from traffic

Instead of writing a rule from scratch, you can click any logged request and turn it into a mock rule. Faultend pre-fills the path, method, and response body from the actual traffic — you just tweak what you need.

This workflow is especially useful during exploratory testing — use your app normally, spot an interesting request in the traffic log, and immediately mock it without having to remember the exact path or response format.

Conditional matching

Beyond path and method, rules can match on request headers, query parameters, body fields, and cookies. This lets you target specific users, sessions, or request variants without affecting all traffic.

Match by header — simulate auth failure for a specific API key
Header: X-Api-Key = test-key-123 → mock 401
Match by query param — break a specific search query
Query param: q contains crash → mock 500
Match by body field — target a specific user ID in a POST request
Body field: userId = 42 → mock 503 with latency

Export & import rule configurations

Save your entire rule set as a JSON file and share it with your team or load it into another server. This is how you replicate a test scenario across environments or version-control your fault injection configs.

# Exported JSON format (from Settings → Export config) { "version": "1", "rules": [ { "name": "Default proxy", "pattern": ".*", "action": "proxy", "target": "https://api.yourapp.com", "priority": 100, "enabled": true }, { "name": "Simulate checkout outage", "pattern": "/api/checkout", "action": "mock", "status": 503, "body": "{\"error\":\"service_unavailable\"}", "priority": 200, "enabled": false } ] }

Find export and import in the server's Settings drawer (gear icon). Import supports two modes: merge (add imported rules to existing ones) and replace (clear existing rules first).

Collaboration

Fault servers can be shared with teammates. Open the Settings drawer and generate an invite link. Anyone with the link can join as a collaborator.

Role What they can do
Owner Full control — manage rules, traffic, settings, collaborators, and delete the server
Admin Manage rules and traffic, generate invite links, promote collaborators. Cannot delete the server or transfer ownership
Collaborator View traffic and rules, toggle rules on/off. Cannot edit rules or manage members
Self-hosting

Run your own instance

Faultend is open source under the MIT licence. You can run a fully private instance on your own infrastructure with no usage limits, no accounts, and no dependency on faultend.com.

You need
→ A Linux VPS (1 vCPU / 512 MB RAM is fine) → Docker + Docker Compose → A domain with wildcard DNS support → Traefik v3 (or any reverse proxy)
You get
→ Unlimited servers and rules → Full data sovereignty → No external dependencies → Optional OAuth (or dev-login for internal tools)
repo

Source code is at github.com/dalogax/faultend. The docker-compose.yml and Dockerfile are included in the repo root.

Prerequisites

Docker ≥ 24
For running the app container and PostgreSQL
Traefik v3
Reverse proxy for HTTPS termination and subdomain routing. Can be replaced with nginx/Caddy if you configure wildcard routing yourself
Wildcard DNS
Both yourdomain.com and *.yourdomain.com must resolve to your server's IP address
PostgreSQL 16
The schema is applied automatically at startup — no manual migration step needed

Installation

01

Clone the repository

git clone https://github.com/dalogax/faultend.git cd faultend
02

Configure environment variables

Copy the example file and edit it. The minimum required variables are shown below. See the Environment variables reference for all options.

cp .env.example .env # edit .env with your values
# Minimum required .env DATABASE_URL=postgresql://faultend:faultend@postgres:5432/faultend SESSION_SECRET=change-this-to-a-64-char-random-string ROOT_DOMAIN=yourdomain.com PORT=3000 NODE_ENV=production SAMPLE_DATA=false

Generate a secure session secret with:

openssl rand -hex 32
03

Update docker-compose.yml with your domain

Edit docker-compose.yml and replace faultend.com with your domain in the Traefik labels:

# Replace these two lines with your domain: - "traefik.http.routers.faultend-http.rule=Host(`yourdomain.com`) || HostRegexp(`^[a-z0-9-]+\\.yourdomain\\.com$`)" - "traefik.http.routers.faultend-https.rule=Host(`yourdomain.com`) || HostRegexp(`^[a-z0-9-]+\\.yourdomain\\.com$`)" # Update the TLS domain: - "traefik.http.routers.faultend-https.tls.domains[0].main=yourdomain.com" - "traefik.http.routers.faultend-https.tls.domains[0].sans=*.yourdomain.com"
04

Start the stack

If you are using Traefik via Docker, make sure the coolify network exists (or rename it to whatever your Traefik network is called).

# Create the external Traefik network if it doesn't exist docker network create coolify # Start Faultend (and PostgreSQL if you add it to compose) docker compose up -d

The app is now reachable through Traefik at https://app.yourdomain.com. The database schema is applied automatically on first startup.

05

Verify health

curl -s https://app.yourdomain.com/health # → {"status":"ok"}
tip

PostgreSQL is not included in the default docker-compose.yml because Coolify manages it separately. For a standalone deployment, add a postgres service to your compose file and set DATABASE_URL accordingly.

# Example: standalone PostgreSQL service to add to docker-compose.yml services: postgres: image: postgres:16-alpine restart: unless-stopped environment: POSTGRES_USER: faultend POSTGRES_PASSWORD: change-this-password POSTGRES_DB: faultend volumes: - postgres_data:/var/lib/postgresql/data volumes: postgres_data:

Environment variables

All configuration is done via environment variables. Copy .env.example and fill in the values for your deployment.

Variable Required Default Description
DATABASE_URL required — PostgreSQL connection string. Format: postgresql://user:pass@host:5432/dbname
SESSION_SECRET required — Random string used to sign session cookies. Generate with openssl rand -hex 32. Must be kept secret.
ROOT_DOMAIN required — Your apex domain (e.g. yourdomain.com or localhost for local dev). Used to build subdomain URLs.
PORT optional 3000 HTTP port the app listens on inside the container.
NODE_ENV optional development Set to production to enable trust-proxy (required when behind Traefik/nginx).
SAMPLE_DATA optional false Set to true to seed three example servers on startup. Useful for demos. Set to false in production.
MOCK_AUTH_ENABLED optional false Set to true to enable the /api/auth/dev-login endpoint that logs in without OAuth. Never enable in production.
GOOGLE_CLIENT_ID optional — Google OAuth 2.0 client ID. Required to enable Google login. See OAuth setup.
GOOGLE_CLIENT_SECRET optional — Google OAuth 2.0 client secret.
GITHUB_CLIENT_ID optional — GitHub OAuth App client ID. Required to enable GitHub login. See OAuth setup.
GITHUB_CLIENT_SECRET optional — GitHub OAuth App client secret.
INITIAL_ADMIN_EMAIL optional — Email address of the first platform admin. On startup, if a user with this email exists in the database, they are granted is_admin=true automatically.
TRAFFIC_RETENTION_DAYS optional 7 Number of days to keep traffic logs before they are automatically deleted. A cleanup job runs every 24 hours.
METRICS_ENABLED optional false Set to true to expose a Prometheus-compatible /metrics endpoint on the app subdomain.
METRICS_TOKEN optional — Bearer token to protect the /metrics endpoint. Recommended in production if metrics are enabled.
POSTHOG_PUBLIC_KEY optional — PostHog project API key for analytics on the landing page. Leave blank to disable analytics entirely.
POSTHOG_HOST optional https://eu.i.posthog.com PostHog ingestion endpoint. Change to the US region or your self-hosted PostHog URL if needed.

Traefik & DNS setup

Faultend uses subdomain routing — app.yourdomain.com for the UI and [server-id].yourdomain.com for each proxy. Both must resolve to the same server, and Traefik must route all subdomains to the Faultend container.

DNS records

# Two DNS records required (at your DNS provider) Type Name Value A yourdomain.com YOUR_SERVER_IP A *.yourdomain.com YOUR_SERVER_IP
note

Wildcard TLS certificates (for *.yourdomain.com) require a DNS-01 ACME challenge. Traefik supports this natively for most DNS providers — configure the certresolver in Traefik's static configuration with your DNS provider's API token.

Traefik architecture

Internet → DNS → Server IP:443 → Traefik → Faultend container:3000 ↓ Routes all *.yourdomain.com → Faultend Faultend reads the Host header to determine whether the request is for app.* (UI/API) or [server-id].* (proxy)

The included docker-compose.yml contains all the Traefik labels you need. Update the domain references and the Traefik network name if yours differs from coolify.

tip

If you use Coolify to manage your server, you can import the repo directly — Coolify will use the docker-compose.yml and Traefik is already set up for you.

OAuth setup

Faultend supports Google and GitHub OAuth for user login. Both are optional — if neither is configured and MOCK_AUTH_ENABLED=true, you can use the dev-login endpoint. For a public-facing instance, configure at least one OAuth provider.

Google OAuth

01

Create a project in Google Cloud Console

Go to console.cloud.google.com, create a project, and enable the Google+ API (or People API).

02

Create OAuth credentials

Under APIs & Services → Credentials, create an OAuth 2.0 Client ID. Set the application type to Web application.

03

Add the redirect URI

https://app.yourdomain.com/api/auth/google/callback
04

Add credentials to .env

GOOGLE_CLIENT_ID=your-client-id.apps.googleusercontent.com GOOGLE_CLIENT_SECRET=your-client-secret

GitHub OAuth

01

Create a GitHub OAuth App

Go to github.com/settings/developers → OAuth Apps → New OAuth App.

02

Set the callback URL

https://app.yourdomain.com/api/auth/github/callback
03

Add credentials to .env

GITHUB_CLIENT_ID=your-github-client-id GITHUB_CLIENT_SECRET=your-github-client-secret

First login & platform admin

After deployment, navigate to https://app.yourdomain.com and sign in with your configured OAuth provider. Your account is created automatically.

To make yourself the platform admin (required to promote users to Pro), set INITIAL_ADMIN_EMAIL to your email address before starting the server. On the next startup, Faultend will grant admin access to that account automatically.

INITIAL_ADMIN_EMAIL=your@email.com

Platform admins can access https://app.yourdomain.com/admin to manage users, view usage, and promote accounts to Pro.

dev only

For local development or internal tools where you don't want to set up OAuth, set MOCK_AUTH_ENABLED=true and visit http://app.yourdomain.com/api/auth/dev-login to log in as a test user. Never enable in production.

Production tips

Secure the session secret
Never commit SESSION_SECRET to the repo. Use environment variables in your deployment platform (Coolify, Render, Fly.io). Generate with openssl rand -hex 32.
Set NODE_ENV=production
This enables Express's trust proxy setting, which is required for correct IP detection and secure session cookies behind Traefik.
Disable SAMPLE_DATA
Set SAMPLE_DATA=false. The sample servers are only useful for exploring the UI locally.
Back up PostgreSQL
All data (users, servers, rules, traffic) lives in PostgreSQL. Set up automated backups. A daily pg_dump piped to a storage bucket is sufficient for most deployments.
Monitor the health endpoint
The app exposes GET /health returning {"status":"ok"}. Plug this into your uptime monitor or Traefik's health check configuration.
Adjust traffic retention
The default is 7 days. Increase or decrease TRAFFIC_RETENTION_DAYS based on your storage capacity and compliance requirements. Be aware that traffic logs may contain request/response bodies.

Updating to a new version

# Pull the latest code git pull origin main # Rebuild and restart the container docker compose build --no-cache docker compose up -d # Verify health curl -s https://app.yourdomain.com/health

The database schema is applied idempotently on every startup — new columns are added with IF NOT EXISTS guards, so updates never drop existing data. You can review the schema at db/schema.sql.
