── Why Faultend

A proxy that inspects, mocks, and breaks your HTTP traffic.

Point your client at a Faultend URL instead of your backend. Every request is logged and can be intercepted with a rule — mock the response, inject a delay, or return a 5xx.
01

Real-time inspection

Every request and response, streamed live. Full visibility into method, path, status, timing, headers, and body — no log diving.

● last updated · just now
02

One-click mocking

Click any logged request to turn it into a mock rule. Edit the status code, response body, and latency — no manual config.

GET/users/42MOCK
03

Multi-backend routing

Match by regex path and HTTP method, ordered by priority. Each rule proxies to its own upstream — perfect for microservices.

^/payments/.* → yoursecondbackend.faultend.com
.*            → yourapp.faultend.com
04

Failure injection

Return any status code, any body. Add fixed delays or random latency in a range. Make a flaky endpoint flaky on demand.

503100–500msMOCK
05

Flexible responses

Compose responses from templates with timestamps, UUIDs, and randoms — or forward to the real upstream and transform what comes back: patch JSON fields, swap status codes, strip headers. Realistic mocks without realistic payloads; edge cases without backend changes.

{ id: {{uuid()}}, ts: {{timestamp()}} }
set $.user.email = null
06

Share & collaborate

Invite teammates to a server with view or edit access. Rule changes apply for everyone — useful for pairing on a flaky bug or handing a repro to QA.

── How it works

Five steps. Zero changes to your app.

~3 min · skim
1

Point your app at Faultend

Create a server, give it an ID, and point your app at the proxy URL we hand back. Every request flows through and gets logged.

PROXY · transparent
API_BASE_URL=https://yourapp.faultend.com
2

Create routing rules

Match each request by HTTP method and a regex path pattern. Rules are evaluated in priority order — the first match wins.

5 RULES · ordered
200 GET ^/users/[0-9]+$ → mock 180 GET ^/products$ → mock 100 * .* → proxy
3

Use your app normally

Watch every call land in the traffic log: method, path, status, time, and whether a rule matched. Filter by method, status, or path.

REC · live stream
GET /users/5 200 250ms ✓ GET /products 500 295ms ✓ DELETE /posts/1 403 9ms ✓
4

Inject failures on demand

Switch any rule to Mock mode. Pick a status code, write a JSON body, and dial in a fixed delay or a random range. Toggle it on or off live.

MOCK · armed
Mock ^/products$ 500 {} Latency range 100–500ms
5

Validate resilience

Watch your app retry, back off, degrade, and recover — and pinpoint exactly which endpoints are making it easy to break. Each rule rolls up into a verdict so you can see what your app survives and what it depends on.

VERDICT · mostly resilient
GET /users/:id PASS resilient GET /products DEGRADE stale cache served POST /orders HARD DEP checkout blocks GET /notifications HARD DEP session screen hangs
